# 系统架构文档

## 版本信息
- 版本: v3.1
- 更新日期: 2026-03-22
- 架构状态: 双模式架构（基础/智伴）

## 1. 系统概述

### 1.1 设计理念
本系统采用**单一真相来源（Single Source of Truth）**架构，以`SystemCore`作为唯一状态管理中心。

### 1.2 双模式架构
```
┌─────────────────────────────────────────────────────────────┐
│                         系统架构                             │
├─────────────────────────────────────────────────────────────┤
│  模式一: 基础模式 (Basic Mode)                               │
│  ├─ AI不参与游戏难度调整                                     │
│  ├─ AI不会主动式交互                                         │
│  ├─ AI不会主动采集数据进行分析                               │
│  └─ AI仅支持页面切换和一问一答功能                           │
│                                                              │
│  模式二: 智伴模式 (Companion Mode)                           │
│  ├─ AI基于数据实现主动式交互                                 │
│  ├─ AI基于状态理解提供健康监测                               │
│  ├─ AI提供认知训练和情感陪伴                                 │
│  └─ 可在设置里选择开启/关闭                                  │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 技术栈
- **后端**: Flask + Flask-SocketIO + Python 3.9+
- **前端**: Vue.js 3 + Nuxt.js
- **通信**: Socket.IO (实时双向通信)
- **感知**: OpenCV + MediaPipe (姿态检测)
- **AI**: Ollama (本地LLM推理)

## 3. 核心模块详解

### 3.1 SystemCore - 系统核心

**文件**: `backend/core/system_core.py`

**职责**:
- 系统唯一状态管理中心
- 线程安全的状态访问和修改
- 状态变更自动广播到所有前端
- 数据持久化（配置和历史记录）

**核心状态结构**:
```python
_state = {
    'aiMode': 'basic',           # AI模式: basic/companion
    'currentPage': '/',          # 当前页面
    'game': {                    # 游戏状态
        'status': 'IDLE',        # IDLE/READY/PLAYING/PAUSED/SETTLING
        'currentGame': None,     # 当前游戏ID
        'difficulty': 3,         # 难度等级
        'dwellTime': 2000,       # 确认时间(ms)
    },
    'gameRuntime': {             # 游戏运行时数据
        'score': 0,
        'timer': 60,
        'accuracy': 0,
    },
    'perception': {              # 感知数据
        'personDetected': False,
        'emotion': 'neutral',
        'attention': 0,
        'fatigue': 0,
        'footPosition': {...},
    },
    'settings': {                # 系统设置
        'dwellTime': 2000,
        'soundEnabled': True,
    },
}
```

**线程安全机制**:
- 使用`RLock`保护状态访问
- 使用`_instance_lock`保护单例创建
- 状态复制后广播，避免竞态条件

---

### 3.2 GameManager - 游戏管理器

**文件**: `backend/games/games_manager.py`

**职责**:
- 游戏注册和创建
- 游戏生命周期管理（准备/开始/暂停/停止）
- 游戏状态同步到SystemCore

**状态流转**:
```
IDLE → READY → PLAYING → SETTLING → IDLE
        ↑       ↓
       STOP   PAUSED
```

---

### 3.3 GameBase - 游戏基类

**文件**: `backend/games/games_base.py`

**职责**:
- 定义游戏通用接口
- 管理游戏状态机
- 提供抽象方法供子类实现

**抽象方法**:
- `_on_ready()`: 准备状态回调
- `_on_start()`: 开始游戏回调
- `_on_update()`: 游戏更新回调（每帧调用）
- `_on_action()`: 玩家动作回调

---

### 3.4 PerceptionManager - 感知管理器

**文件**: `backend/perception/perception_manager.py`

**职责**:
- 整合所有检测（人脸、情绪、心率等）
- 处理摄像头帧数据
- 提供用户状态摘要

**注意**: 当前实现为简化版，实际可以接入深度学习模型。

---

### 3.5 ScreenProcessor - 投影处理器

**文件**: `backend/perception/perception_screen_processor.py`

**职责**:
- 投影摄像头处理
- 脚部位置检测（MediaPipe Pose）
- 透视变换校准
- 区域检测

## 4. 通信协议

### 4.1 Socket.IO 事件

**前端 → 后端**:
```javascript
// 页面导航
socket.emit('navigate', { page, source })

// 游戏控制
socket.emit('game_control', { action, game, ... })
// action: 'ready', 'start', 'pause', 'stop', 'restart'

// 设置AI模式
socket.emit('set_ai_mode', { mode })

// 设置确认时间
socket.emit('set_dwell_time', { dwellTime })
```

**后端 → 前端**:
```javascript
// 系统状态广播
socket.emit('system_state', state)

// 感知数据更新
socket.emit('perception_update', perception_data)

// 游戏状态更新
socket.emit('game_update', game_state)

// 页面跳转指令
socket.emit('navigate_to', { page })

// AI说话
socket.emit('akon_speak', { text })
```

### 4.2 HTTP API

```
GET  /api/status              # 获取系统状态
GET  /api/game/list          # 获取游戏列表
POST /api/akon/chat          # 阿康对话
GET  /api/perception/feed    # 摄像头视频流
```

## 5. 游戏状态机

### 5.1 状态定义
```python
class GameStatus(Enum):
    IDLE = "IDLE"           # 空闲
    READY = "READY"         # 准备中
    PLAYING = "PLAYING"     # 游戏中
    PAUSED = "PAUSED"       # 暂停
    SETTLING = "SETTLING"   # 结算中
```

### 5.2 状态流转图
```
                    ┌─────────────┐
                    │    IDLE     │
                    └──────┬──────┘
                           │ set_ready()
                           ▼
                    ┌─────────────┐
         ┌─────────│    READY    │◄────────┐
         │         └──────┬──────┘         │
         │                │ start_game()   │ restart()
         │                ▼                │
  stop() │         ┌─────────────┐         │
         │    ┌───│   PLAYING   │───┐     │
         │    │   └──────┬──────┘   │     │
         │    │pause()   │          │     │
         │    ▼          │          │     │
         │  ┌────────┐   │          │     │
         └──│ PAUSED │───┘          │     │
            └────┬───┘              │     │
                 └──────────────────┘     │
                           │ timeout       │
                           ▼               │
                    ┌─────────────┐        │
                    │  SETTLING   │────────┘
                    └──────┬──────┘
                           │ settle complete
                           ▼
                    ┌─────────────┐
                    │    IDLE     │
                    └─────────────┘
```

## 7. 改进建议

### 7.1 架构层面

1. **引入事件总线**: 使用发布-订阅模式解耦模块
2. **状态管理优化**: 考虑使用Redux或Pinia模式
3. **配置管理**: 使用专门的配置管理模块，支持环境变量
4. **日志系统**: 统一使用Python logging模块

### 7.2 代码层面

1. **重构长函数**: 将超过50行的函数拆分
2. **统一错误处理**: 使用异常类而不是返回False/None
3. **添加单元测试**: 为核心模块添加测试
4. **代码格式化**: 使用Black或Ruff统一代码风格

### 7.3 性能层面

1. **状态广播优化**: 使用增量更新而不是全量广播
2. **感知数据处理**: 降低处理频率或使用更高效算法
3. **前端渲染**: 使用虚拟滚动或节流

## 9. 总结

### 9.1 架构优点
1. **单一真相来源**: SystemCore作为唯一状态中心，逻辑清晰
2. **双模式设计**: 基础/智伴模式满足不同需求
3. **游戏状态机**: 清晰的状态流转，易于扩展新游戏
4. **实时通信**: Socket.IO提供良好的实时体验

### 9.2 主要风险
1. **线程安全**: 部分全局状态缺少锁保护
2. **错误处理**: 过于宽泛的异常捕获可能隐藏问题
3. **代码复杂度**: 部分函数过长，职责不清晰
4. **配置管理**: 硬编码配置过多，维护困难
