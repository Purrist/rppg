
## 五、游戏流程设计

### 5.1 整体游戏流程

#### 5.1.1 游戏启动流程

```
用户操作流程：

1. 用户在平板端进入"益智"页面（learning.vue）
   │
   ▼
2. 浏览游戏列表，点击"处理速度训练"卡片
   │
   ▼
3. 弹出游戏信息弹窗，显示：
   - 游戏名称：处理速度训练
   - 游戏说明：提高反应速度和认知灵活性
   - 训练模块：Go/No-Go、选择反应、序列反应
   - 预计时长：约5分钟
   │
   ▼
4. 用户点击"开始游戏"
   │
   ├── 从localStorage读取用户设置的确认时间
   │
   ▼
5. 发送游戏控制指令：
   socket.emit('game_control', {
     action: 'ready',
     game: 'processing_speed',
     dwell_time: 2000  // 用户设置的确认时间
   })
   │
   ▼
6. 后端创建游戏实例，进入READY状态
   │
   ▼
7. 前端跳转到training.vue（平板端）和projection.vue（投影端）
   │
   ▼
8. 投影端显示准备圆圈，等待用户进入
```

### 5.2 单题流程详解

#### 5.2.1 Go/No-Go模块单题流程

```
Go/No-Go单题流程：

时间轴：
0ms        1000-2000ms      3000-8000ms        确认时间后      确认时间+1000ms
│              │                 │                 │                 │
│   间隔期     │    题目呈现     │    用户响应     │     反馈       │   下一题
│   (空白)     │   (刺激显示)    │   (移动+停留)   │   (对错显示)   │   (间隔)
│              │                 │                 │                 │
▼              ▼                 ▼                 ▼                 ▼

详细流程：

1. 间隔期（1000-2000ms）
   - 所有区域显示为灰色
   - 无进度环响应
   - 屏幕显示"准备..."

2. 题目呈现（作答时间开始）
   - 随机选择1-4个区域激活
   - Go条件：目标区域显示绿色
   - No-Go条件：目标区域显示红色
   - 屏幕显示指令："踩绿色！" 或 "别踩红色！"
   - 作答倒计时条开始递减

3. 用户响应
   a) Go条件（绿色区域）：
      - 用户移动到目标区域
      - 进度环开始增长
      - 停留满确认时间后触发确认
      - 判定为正确，显示绿色✓
   
   b) No-Go条件（红色区域）：
      - 用户应保持不动或离开区域
      - 作答时间结束仍未踩踏
      - 判定为正确，显示"抑制成功"
   
   c) 错误情况：
      - Go条件未踩踏：超时错误
      - No-Go条件踩踏：抑制失败

4. 反馈显示（1000ms）
   - 正确：绿色✓ + "+10分"
   - 错误：红色✗ + "-5分"
   - 超时：灰色 + "时间到"
   - 显示反应时间

5. 进入下一题间隔期
```

```
   - 每个激活区域显示不同颜色

   - 停留满确认时间后触发确认

4. 反馈显示
   - 正确：绿色✓ + "+10分" + 反应时间
   - 错误：红色✗ + "-5分" + 正确答案提示
   - 超时：灰色 + "时间到"

```

```

   - 完成整个序列：额外奖励 +50分

4. 序列重复
   - 同一序列重复5次
   - 观察反应时间是否缩短（内隐学习指标）
```

## 六、游戏规则详解

### 6.1 核心概念定义

#### 6.1.1 题目定义

**题目**是游戏的基本单位，每道题包含以下要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| 刺激 | 呈现给用户的视觉信息 | 绿色区域、红色区域、颜色指令 |
| 选项 | 用户可选择的目标 | 8个识别区域 |
| 正确答案 | 题目的正确响应 | 特定区域或"不响应" |
| 作答时间 | 完成答题的时间限制 | 3000-8000ms |
| 分值 | 正确响应的得分 | +10分（基础分） |

#### 6.1.2 确认时间定义

**确认时间**是用户在区域内停留以确认选择所需的时间：

```
确认时间特性：

1. 固定性
   - 确认时间在单次游戏过程中保持不变
   - 不参与难度调整
   - 仅可在设置页面调整

2. 可调范围
   - 最小值：1500ms（1.5秒）
   - 默认值：2000ms（2秒）
   - 最大值：4000ms（4秒）

3. 设计原理
   - 防止误触：足够长的停留时间确保意图明确
   - 适应老年人：考虑老年人反应和移动速度
   - 用户控制：用户可根据自身情况调整

4. 与作答时间的关系
   - 作答时间 ≥ 确认时间 + 1000ms
   - 确保用户有足够时间完成确认
```

#### 6.1.3 作答时间定义

**作答时间**是每道题的答题时限：

```
作答时间特性：

1. 动态性
   - 作答时间根据难度等级动态调整
   - 是难度调整的主要维度

2. 计算公式
   作答时间 = 确认时间 + 难度时间增量
   
   其中难度时间增量为1000-5000ms

3. 视觉表现
   - 作答倒计时条：从绿色逐渐变为红色
   - 时间紧迫时闪烁提示

4. 超时处理
   - 超时视为答题失败
   - 扣除5分
   - 进入下一题
```

#### 6.1.4 作答间隔定义

**作答间隔**是两道题之间的休息时间：

```
作答间隔特性：

1. 时长
   - 默认：1500ms
   - 范围：1000-2000ms
   - 可根据难度调整

2. 期间行为
   - 所有区域显示为灰色
   - 用户进入任何区域不会触发进度环
   - 屏幕显示"准备..."

3. 设计目的
   - 提供短暂休息
   - 准备下一题
   - 避免连续刺激导致的疲劳
```

#### 6.1.5 干扰项定义

**干扰项**是增加任务难度的非目标元素：

```
干扰项类型：

1. No-Go刺激（Go/No-Go模块）
   - 红色区域，需要抑制响应
   - 增加认知冲突

2. 相似颜色（选择反应模块）
   - 与目标颜色相似的其他颜色
   - 增加视觉搜索难度

3. 多激活区域
   - 同时激活多个区域
   - 增加选择难度

干扰项数量与难度关系：
- 低难度（1-2级）：无干扰项
- 中难度（3-5级）：1-2个干扰项
- 高难度（6-8级）：3-4个干扰项
```

### 6.2 计分规则

#### 6.2.1 基础计分

| 情况 | 得分 | 说明 |
|------|------|------|
| Go正确 | +10分 | 正确踩踏绿色区域 |
| No-Go正确 | +15分 | 正确抑制不踩红色区域 |
| 选择反应正确 | +10分 | 正确选择目标颜色 |
| 序列反应正确（每步） | +10分 | 正确完成序列的一步 |
| 序列完成奖励 | +50分 | 正确完成整个序列 |
| Go错误 | -5分 | 未踩踏绿色区域 |
| No-Go错误 | -10分 | 错误踩踏红色区域 |
| 选择反应错误 | -5分 | 选择错误颜色 |
| 序列反应错误 | -5分 | 序列步骤错误 |
| 超时 | -5分 | 未在作答时间内完成 |

#### 6.2.2 反应时间奖励

```
反应时间奖励计算：

基础分 × (1 + 时间奖励系数)

时间奖励系数 = (作答时间 - 实际反应时间) / 作答时间

示例：
- 作答时间：5000ms
- 实际反应时间：2000ms
- 时间奖励系数 = (5000 - 2000) / 5000 = 0.6
- 最终得分 = 10 × (1 + 0.6) = 16分

上限：基础分 × 2（最高翻倍）
```

#### 6.2.3 连续正确奖励

```
连续正确奖励：

连续正确次数    奖励倍数
    3次          ×1.2
    5次          ×1.5
    10次         ×2.0
    15次以上      ×2.5

示例：
- 连续正确5次后，第6次正确得分 = 10 × 1.5 = 15分
- 连续正确中断后，奖励倍数重置
```

### 6.3 各模块规则详解

#### 6.3.1 Go/No-Go模块规则

**任务目标**：训练反应抑制能力

**规则说明**：

```
Go条件（需要响应）：
1. 目标区域显示绿色
2. 屏幕显示"踩绿色！"
3. 用户需要在作答时间内踩踏绿色区域
4. 停留确认时间后判定为正确
5. 得分：+10分 + 反应时间奖励

No-Go条件（需要抑制）：
1. 目标区域显示红色
2. 屏幕显示"别踩红色！"
3. 用户需要保持不动或离开区域
4. 作答时间结束未踩踏判定为正确
5. 得分：+15分（抑制成功的额外奖励）

错误情况：
1. Go条件未响应：超时错误，-5分
2. No-Go条件踩踏：抑制失败，-10分
3. 踩踏错误区域：选择错误，-5分
```

**Go/No-Go比例**：

| 难度等级 | Go比例 | No-Go比例 | 说明 |
|----------|--------|-----------|------|
| 1-2级 | 80-90% | 10-20% | Go占多数，抑制需求低 |
| 3-5级 | 60-70% | 30-40% | 平衡的Go/No-Go比例 |
| 6-8级 | 40-50% | 50-60% | 高抑制需求 |

#### 6.3.2 选择反应模块规则

**任务目标**：训练决策速度和视觉搜索能力

**规则说明**：

```
基本流程：
1. 屏幕显示目标颜色指令："踩蓝色！"
2. 2-8个区域被激活，显示不同颜色
3. 用户需要找到并踩踏目标颜色区域
4. 停留确认时间后判定
5. 正确：+10分 + 反应时间奖励
6. 错误：-5分

颜色配置：
- 低难度：2-3种颜色，差异明显
- 中难度：4-5种颜色，部分相似
- 高难度：6-8种颜色，多组相似色

相似颜色组：
- 红色系：红色、橙色、粉色
- 蓝色系：蓝色、青色、紫色
- 绿色系：绿色、黄绿色
- 黄色系：黄色、金色
```

**选择数量与难度**：

| 难度等级 | 激活区域数 | 颜色种类 | 干扰项 |
|----------|------------|----------|--------|
| 1-2级 | 2-3个 | 2-3种 | 无 |
| 3-4级 | 4-5个 | 4-5种 | 1-2个相似色 |
| 5-6级 | 6-7个 | 5-6种 | 2-3个相似色 |
| 7-8级 | 8个 | 6-8种 | 3-4个相似色 |

#### 6.3.3 序列反应模块规则

**任务目标**：训练序列学习和程序记忆

**规则说明**：

```
基本流程：
1. 屏幕显示序列提示："2 → 5 → 3"
2. 用户按顺序踩踏区域2、区域5、区域3
3. 每步需要停留确认
4. 显示序列进度："1/3"、"2/3"、"3/3"
5. 完成整个序列后获得额外奖励

序列长度：
- 低难度：3步序列
- 中难度：4-5步序列
- 高难度：6-8步序列

序列类型：
1. 显式序列
   - 屏幕显示完整序列
   - 用户按显示顺序执行

2. 隐式序列
   - 仅显示当前目标区域
   - 用户需要记住序列模式
   - 同一序列重复5次

计分规则：
- 每步正确：+10分
- 完成序列奖励：+50分
- 步骤错误：序列重置，-5分
```

**序列学习评估**：

```
内隐学习指标：
- 同一序列重复5次
- 记录每次的反应时间
- 反应时间逐渐缩短表明内隐学习发生

学习效果计算：
学习效果 = (第1次平均RT - 第5次平均RT) / 第1次平均RT × 100%

示例：
- 第1次平均RT：2500ms
- 第5次平均RT：1800ms
- 学习效果 = (2500 - 1800) / 2500 × 100% = 28%
```

## 七、难度分级标准

### 7.1 难度维度定义

基于文献综述和游戏设计，处理速度训练游戏的难度通过以下维度进行调整：

#### 7.1.1 时间维度

| 参数 | 低难度 | 中难度 | 高难度 |
|------|--------|--------|--------|
| 作答时间 | 7000-8000ms | 4000-6000ms | 3000-3500ms |
| 作答间隔 | 2000ms | 1500ms | 1000ms |
| 反馈时长 | 1500ms | 1000ms | 500ms |

#### 7.1.2 刺激维度

| 参数 | 低难度 | 中难度 | 高难度 |
|------|--------|--------|--------|
| 激活区域数 | 1-2个 | 3-5个 | 6-8个 |
| 颜色种类 | 2-3种 | 4-5种 | 6-8种 |
| 刺激相似度 | 低 | 中 | 高 |

#### 7.1.3 任务维度

| 参数 | 低难度 | 中难度 | 高难度 |
|------|--------|--------|--------|
| Go/No-Go比例 | 80-90% Go | 60-70% Go | 40-50% Go |
| 序列长度 | 3步 | 4-5步 | 6-8步 |
| 干扰项数量 | 0个 | 1-2个 | 3-4个 |

### 7.2 八级难度标准

基于上述难度维度，设计了8级难度标准：

#### 7.2.1 难度等级1（最简单）

```
难度等级1参数：

Go/No-Go模块：
- Go比例：90%
- 作答时间：8000ms
- 激活区域数：1个
- 干扰项：无

选择反应模块：
- 激活区域数：2个
- 颜色种类：2种（差异明显）
- 作答时间：8000ms
- 干扰项：无

序列反应模块：
- 序列长度：3步
- 作答时间：8000ms
- 序列类型：显式序列

适用人群：
- 初次训练者
- 认知功能较弱者
- 需要建立信心的用户
```

#### 7.2.2 难度等级2

```
难度等级2参数：

Go/No-Go模块：
- Go比例：80%
- 作答时间：7000ms
- 激活区域数：1个
- 干扰项：无

选择反应模块：
- 激活区域数：2个
- 颜色种类：3种
- 作答时间：7000ms
- 干扰项：无

序列反应模块：
```

## 八、基于作答情况的难度调整策略

### 8.1 调整策略理论基础

#### 8.1.1 最佳训练区间理论

根据García-Betancourt et al.（2023）的研究，认知训练的最佳效果发生在正确率60-90%的区间内。这一区间被称为"最佳训练区间"（Optimal Training Zone）：

```
正确率区间与训练效果：

正确率 < 60%：
- 任务过难
- 用户容易产生挫败感
- 训练效果下降
- 应降低难度

正确率 60-90%：
- 最佳训练区间
- 任务难度适中
- 用户保持参与动机
- 训练效果最佳

正确率 > 90%：
- 任务过易
- 用户可能感到无聊
- 训练效果下降
- 应提高难度
```

#### 8.1.2 自适应调整原则

1. **渐进性原则**：难度调整应渐进进行，每次只调整一个等级
2. **稳定性原则**：避免频繁调整，需要足够的观察窗口
3. **个体化原则**：调整策略应考虑个体差异
4. **响应性原则**：对用户表现变化做出及时响应

### 8.2 表现评估指标

#### 8.2.1 核心指标定义

| 指标 | 计算方法 | 权重 |
|------|----------|------|
| 正确率 | 正确次数 / 总次数 × 100% | 50% |
| 平均反应时间 | 所有正确响应的反应时间平均值 | 30% |
| 反应时间稳定性 | 反应时间标准差 / 平均反应时间 | 10% |
| 连续正确数 | 当前连续正确的次数 | 10% |

#### 8.2.2 综合表现分数

```
综合表现分数计算：

Performance Score = 
    0.5 × 正确率标准化分数 +
    0.3 × 反应时间标准化分数 +
    0.1 × 稳定性标准化分数 +
    0.1 × 连续正确标准化分数

其中：
- 正确率标准化分数 = (正确率 - 50%) / 50% × 100
- 反应时间标准化分数 = (基准RT - 实际RT) / 基准RT × 100
- 稳定性标准化分数 = (1 - 变异系数) × 100
- 连续正确标准化分数 = min(连续正确数 / 10, 1) × 100
```

### 8.3 难度调整算法

#### 8.3.1 基于滑动窗口的评估

```
滑动窗口评估机制：

窗口大小：5个试次
评估频率：每完成5个试次评估一次

窗口内数据：
- 正确次数
- 错误次数
- 平均反应时间
- 反应时间标准差

评估公式：
窗口正确率 = 窗口内正确次数 / 5
窗口平均RT = 窗口内正确响应RT的平均值
```

#### 8.3.2 难度调整规则

```
难度调整决策树：

评估窗口正确率
    │
    ├── 正确率 < 60%
    │       │
    │       ├── 连续2个窗口 < 60%？
    │       │       │
    │       │       ├── 是 → 降低1个难度等级
    │       │       │
    │       │       └── 否 → 维持当前难度
    │       │
    │       └── 维持当前难度（观察）
    │
    ├── 正确率 60-90%
    │       │
    │       └── 维持当前难度（最佳区间）
    │
    └── 正确率 > 90%
            │
            ├── 平均RT < 基准RT × 0.8？
            │       │
            │       ├── 是 → 提高1个难度等级
            │       │
            │       └── 否 → 维持当前难度
            │
            └── 连续3个窗口 > 90%？
                    │
                    ├── 是 → 提高1个难度等级
                    │
                    └── 否 → 维持当前难度
```

#### 8.3.3 调整冷却机制

```
调整冷却机制：

目的：防止频繁调整导致的体验不稳定

冷却时间：
- 升级冷却：完成10个试次后才能再次升级
- 降级冷却：完成5个试次后才能再次降级

冷却期间：
- 继续收集表现数据
- 不执行难度调整
- 记录调整建议，冷却结束后执行
```

### 8.4 详细调整策略

#### 8.4.1 升级策略

```
升级条件（需同时满足）：

1. 正确率条件
   - 连续3个窗口正确率 > 85%
   - 或连续5个窗口正确率 > 80%

2. 反应时间条件
   - 平均反应时间 < 基准RT × 0.85
   - 基准RT根据当前难度等级确定

3. 稳定性条件
   - 反应时间变异系数 < 0.3
   - 表现稳定，非偶然高分

4. 冷却条件
   - 距离上次升级已过10个试次

升级执行：
- 难度等级 +1
- 更新基准RT
- 重置冷却计时器
- 显示难度提升提示
```

#### 8.4.2 降级策略

```
降级条件（满足任一）：

1. 严重困难
   - 连续2个窗口正确率 < 50%
   - 立即降级

2. 持续困难
   - 连续3个窗口正确率 < 60%
   - 降级

3. 反应时间过长
   - 平均反应时间 > 作答时间 × 0.8
   - 且正确率 < 70%
   - 降级

降级执行：
- 难度等级 -1
- 不更新基准RT（保留历史基准）
- 重置冷却计时器
- 显示难度降低提示（可选）
```

### 8.6 调整策略实现伪代码

```python
class DifficultyAdjuster:
    """难度调整器"""
    
    def __init__(self):
        self.current_level = 3  # 初始难度等级
        self.window_size = 5    # 评估窗口大小
        self.trial_history = [] # 试次历史
        self.last_adjustment = 0 # 上次调整的试次编号
        self.upgrade_cooldown = 10  # 升级冷却
        self.downgrade_cooldown = 5 # 降级冷却
        
        # 各难度等级的基准RT
        self.baseline_rt = {
            1: 5000, 2: 4500, 3: 4000, 4: 3500,
            5: 3000, 6: 2800, 7: 2500, 8: 2200
        }
    
    def record_trial(self, correct: bool, rt: float):
        """记录试次结果"""
        self.trial_history.append({
            'correct': correct,
            'rt': rt,
            'trial_num': len(self.trial_history) + 1
        })
        
        # 每5个试次评估一次
        if len(self.trial_history) % self.window_size == 0:
            self.evaluate_and_adjust()
    
    def evaluate_and_adjust(self):
        """评估并调整难度"""
        # 获取最近窗口的数据
        recent = self.trial_history[-self.window_size:]
        
        # 计算窗口指标
        accuracy = sum(1 for t in recent if t['correct']) / self.window_size
        avg_rt = sum(t['rt'] for t in recent if t['correct']) / max(sum(1 for t in recent if t['correct']), 1)
        
        # 获取历史窗口数据
        if len(self.trial_history) >= self.window_size * 3:
            prev_window1 = self.trial_history[-self.window_size*2:-self.window_size]
            prev_window2 = self.trial_history[-self.window_size*3:-self.window_size*2]
            
            prev_acc1 = sum(1 for t in prev_window1 if t['correct']) / self.window_size
            prev_acc2 = sum(1 for t in prev_window2 if t['correct']) / self.window_size
        else:
            prev_acc1 = prev_acc2 = accuracy
        
        # 调整决策
        trial_num = len(self.trial_history)
        
        # 升级判断
        if self.should_upgrade(accuracy, avg_rt, prev_acc1, prev_acc2, trial_num):
            self.upgrade()
        
        # 降级判断
        elif self.should_downgrade(accuracy, avg_rt, prev_acc1, prev_acc2, trial_num):
            self.downgrade()
    
    def should_upgrade(self, acc, rt, prev1, prev2, trial_num):
        """判断是否应该升级"""
        # 冷却检查
        if trial_num - self.last_adjustment < self.upgrade_cooldown:
            return False
        
        # 最高难度检查
        if self.current_level >= 8:
            return False
        
        # 正确率条件
        acc_condition = (acc > 0.85 and prev1 > 0.85 and prev2 > 0.85) or \
                       (acc > 0.80 and prev1 > 0.80 and prev2 > 0.80 and 
                        prev1 > 0.80 and prev2 > 0.80)
        
        # 反应时间条件
        rt_condition = rt < self.baseline_rt[self.current_level] * 0.85
        
        return acc_condition and rt_condition
    
    def should_downgrade(self, acc, rt, prev1, prev2, trial_num):
        """判断是否应该降级"""
        # 冷却检查
        if trial_num - self.last_adjustment < self.downgrade_cooldown:
            return False
        
        # 最低难度检查
        if self.current_level <= 1:
            return False
        
        # 严重困难
        if acc < 0.50 and prev1 < 0.50:
            return True
        
        # 持续困难
        if acc < 0.60 and prev1 < 0.60 and prev2 < 0.60:
            return True
        
        # 反应时间过长
        if rt > self.baseline_rt[self.current_level] * 1.5 and acc < 0.70:
            return True
        
        return False
    
    def upgrade(self):
        """执行升级"""
        self.current_level = min(8, self.current_level + 1)
        self.last_adjustment = len(self.trial_history)
        self.emit_adjustment('upgrade', self.current_level)
    
    def downgrade(self):
        """执行降级"""
        self.current_level = max(1, self.current_level - 1)
        self.last_adjustment = len(self.trial_history)
        self.emit_adjustment('downgrade', self.current_level)
    
    def emit_adjustment(self, direction, new_level):
        """发送调整通知"""
        print(f"[难度调整] {direction} -> 等级{new_level}")
        # 发送事件到前端
        # socketio.emit('difficulty_change', {
        #     'direction': direction,
        #     'new_level': new_level,
        #     'params': self.get_level_params(new_level)
        # })
    
    def get_level_params(self, level):
        """获取指定难度等级的参数"""
        params = {
            1: {'go_ratio': 0.90, 'answer_time': 8000, 'zones': 1, 'colors': 2, 'seq_len': 3},
            2: {'go_ratio': 0.80, 'answer_time': 7000, 'zones': 1, 'colors': 3, 'seq_len': 3},
            3: {'go_ratio': 0.70, 'answer_time': 6000, 'zones': 2, 'colors': 3, 'seq_len': 4},
            4: {'go_ratio': 0.60, 'answer_time': 5000, 'zones': 2, 'colors': 4, 'seq_len': 4},
            5: {'go_ratio': 0.50, 'answer_time': 4500, 'zones': 3, 'colors': 5, 'seq_len': 5},
            6: {'go_ratio': 0.50, 'answer_time': 4000, 'zones': 3, 'colors': 5, 'seq_len': 6},
            7: {'go_ratio': 0.40, 'answer_time': 3500, 'zones': 4, 'colors': 6, 'seq_len': 7},
            8: {'go_ratio': 0.40, 'answer_time': 3000, 'zones': 4, 'colors': 8, 'seq_len': 8},
        }
        return params.get(level, params[3])
```
